# 百工 Baigong 🏮

**让 AI Agent 像工匠一样协作的开源框架。**

百工出《周礼》——"百工饬力，各司其职"。每一个 Agent 就像一位工匠：有自己的角色、工具和记忆，通过看板和消息专注协作，在一次次任务中自然成长。

---

## 为什么是百工，而不是又一个工作流编排？

现在的 Agent 框架大多在"编排"——你画好图，Agent 按步骤跑。百工不是这样。百工的组织方式是**协作**，不是编排。

| 工作流编排 | 百工 |
|-----------|------|
| 你画好流程图，Agent 按步骤执行 | 你下任务，Agent 自己想办法完成 |
| 一个中心调度器在控制一切 | 没有调度器，每个 Agent 自主决定做什么 |
| 所有 Agent 共享同一个上下文 | 每个 Agent 有独立的角色、工具、记忆 |
| 流程是静态的，改流程要改代码 | 主管运行时动态拆解任务，灵活应变 |
| 做了就忘，下次再来一遍 | 每次任务自动巩固记忆，越干越熟练 |

在百工里，你看不到"编排"的影子。你看到的是：

**主管看到任务 → 拆解 → 写上看板 → 研究员看到 → 主动领取 → 搜索 → 写结果 → 编辑看到 → 整理去重 → 管理员入库 → 主管收齐汇报**

没有人在中间转消息。没有 DAG。每个人自己看板、自己领、自己干。

## 快速开始

```bash
pip install baigong
```

### 30 秒跑起来

```python
import asyncio
from baigong.core import init_db, MessageBus, TaskBoard, AgentManager, ToolRegistry
from baigong.tool import BaseTool
from baigong.role import RoleDefinition

# 1. 定义你自己的工具
class MySearch(BaseTool):
    ...

# 2. 初始化系统
init_db()
bus, board = MessageBus(), TaskBoard()
manager = AgentManager(bus, board)

# 3. 注册角色、创建 Agent、下发任务
manager.register_role(RoleDefinition(role_id="researcher", ...))
researcher = await manager.create_agent("王研究员", "researcher")
await manager.start_agent(researcher.agent_id)

task = board.create_task(goal="搜索翡翠鉴别资料")
```

完整示例见 [`example.py`](example.py)，100 行跑通"搜索→整理→入库"完整流程。

## 五大核心组件

### 1️⃣ Agent 注册表 — 花名册

每个 Agent 是独立配置的个体——模型、工具、权限、角色，全部独立。甚至可以混用：

- **主管** → DeepSeek（强推理，便宜）
- **研究员** → Claude（长文本好）
- **编辑** → 本地 GGUF（完全免费）

```python
researcher = await manager.create_agent("王研究员", "researcher",
    model_config=ModelConfig(provider="anthropic", model="claude-sonnet-4"),
    tools=["web_search", "read_file", "send_message"],
)
```

### 2️⃣ 消息总线 — 内网通信

Agent 之间通过消息沟通，不是函数调用。支持单播、广播、多轮对话。

```
研究员 → 发给主管: "任务完成，找到 10 篇资料"
主管 → 发给编辑: "可以开始整理了，资料在看板上"
编辑 → 发给主管: "整理完毕，8 篇已分类"
```

### 3️⃣ 任务看板 — 工单

所有信息公开在任务卡片上，Agent 自己去板上取。任务流转是**主管在运行时动态决定**的——看到 A 完成，可能觉得还需细化，再派一个子任务给 A。

### 4️⃣ Agent 运行时 — 工位

每个 Agent 永久运行一个**思考-行动循环**：

```
感知(看板+信箱) → 思考(LLM) → 决策(权限检查) → 行动(工具/消息) → 更新状态
```

| 状态 | 行为 | 巡检频率 |
|------|------|---------|
| 🟢 空闲 | 等新任务 | 每 5 秒 |
| 🟡 工作中 | 专注于当前任务 | — |
| 🔵 等待 | 等别人的结果 | 每 10-15 秒 |
| 🔴 错误/卡住 | 告警 CEO | — |

### 5️⃣ 记忆系统 — 大脑

#### 四层记忆架构

| 层级 | 容量 | 持久化 | 像人脑的哪个部分 |
|------|------|--------|----------------|
| 工作记忆 | ~10 条 | 内存 | "我正在干啥" |
| 短期记忆 | ~100 条 | 内存 + SQLite | "刚发生了什么" |
| 长期记忆 | ∞ | SQLite 树+图 | "我学到了什么" |
| 团队知识 | ∞ | 共享存储 | 大家都知道的常识 |

#### 记忆自动生长，不需要人喂

每次任务完成后，Agent 自动判断这段经历值不值得记住：

**信不信由你，这些信号完全自己判断：**

| 信号 | 显著性 |
|------|--------|
| CEO 给了反馈 | ⭐⭐⭐ 最重要 |
| 任务做失败了 | ⭐⭐ 教训最深刻 |
| 发现了新概念 | ⭐⭐ |
| 已有知识和新结果矛盾了 | ⭐⭐ |
| 类似任务出现了 3 次以上 | ⭐ 模式要提炼了 |

#### 联想式检索 —— 像人脑一样"想到"

想到"翡翠"时，自动浮现"鉴别 → B货 → 证书"。

```
入口: "翡翠"
  ├── 子节点: "翡翠鉴别"（A货、B货、证书）
  ├── 链接: ↔ "百度百科"（最佳来源）
  └── 链接: ↔ "关键词搜索经验"
```

Agent 就是通过这种方式**越干越聪明**的。不是模型变强了，是记忆网络变密了。

## Skill 自动形成

同类型任务做 3 次以上，百工自动从记忆树中挖掘可复用的 **Skill**。

**下次遇到同样任务** → 直接执行 Skill → 跳过 LLM 思考 → 更快更稳。

每次执行成功 → 置信度上升 → 用得越多，越熟练。就像工匠的手艺，越练越精。

## 五层容错 —— Agent 没那么脆弱

| 层级 | 问题 | 怎么做 |
|------|------|--------|
| 1️⃣ | LLM 调用失败 | 重试 → 指数退避 → 熔断 |
| 2️⃣ | 死循环 | 同一工具反复调同一参数 → 打断 |
| 3️⃣ | 任务执行异常 | 自动重试 → 降级 → 通知 CEO |
| 4️⃣ | Agent 崩溃 | 15 秒健康检查 → 自动重启 |
| 5️⃣ | 系统崩溃 | 恢复后重建任务、检查记忆完整性 |

## 开源友好设计

百工不会绑定你的个人环境：

- **只给接口，不给实现** — 自带全是抽象基类，没有搜刮你的 API Key
- **示例模板** — `examples/my_tools.py` 照着写就行
- **YAML 配置** — 角色、Agent、工具全部 YAML 定义

你的私有工具和密钥在外面，不进入仓库。

```
my_company_setup/
├── my_tools.py        # 你的搜索、读文件、调 API
├── my_llm_client.py   # 你的 DeepSeek / Claude 接入
├── roles.yaml         # 你的角色定义
└── config.yaml        # 你的 Agent 配置
```

## 自己做主

1. **继承 `BaseTool`** — 实现你需要的工具
2. **继承 `BaseLLMClient`** — 接上你的模型
3. **写 YAML 定义角色** — 研究员、编辑、主管...
4. **创建 Agent 开工** — `create_agent` → `start_agent` → 下任务
